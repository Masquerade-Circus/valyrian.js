# 5.2. Pulses (`valyrian.js/pulses`)

Pulses give you fine-grained reactivity with a pulse-specific state model.

Use this page as a behavior guide first and an API reference second.

## 5.2.1. When to Use What

Choose the smallest tool that keeps update timing obvious:

| Tool               | Use it for                                                                                                                   | Prefer something else when                                                                                    |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `createPulse`      | One value or a small group of independent values, especially when components can subscribe directly by reading inside render | You want one shared object with named actions, immutable external reads, or batched async transitions         |
| `createPulseStore` | Shared object state with pulse methods, immutable external reads, and debounced subscriber delivery                          | You want a different state model than pulses provide                                                          |
| `FluxStore`        | If you are comparing pulses with Valyrian's other state module, continue with [./5.3-flux-store.md](./5.3-flux-store.md)     | This page is the one you want when you are working with pulse-style reads, writes, batching, and pulse events |

Rule of thumb:

- Start with `createPulse` when the update story is mostly "read in render, write directly".
- Move to `createPulseStore` when several values must move together behind named pulse methods.
- If you are evaluating Valyrian's other state module, continue with [./5.3-flux-store.md](./5.3-flux-store.md); this page stays focused on pulse behavior.

## 5.2.2. What is a Pulse?

In Valyrian, a pulse is a reactive update unit.

That means a pulse does not only hold state. It also defines how that state change is published to subscribers.

- `createPulse` gives you one reactive value with direct reads, writes, and notifications.
- `createPulseStore` gives you a reactive state object plus named pulse methods that can be synchronous or asynchronous.
- `createMutableStore` keeps the same pulse method model, but removes the immutability guard from public state.

For plain values, `createPulse` is the closest comparison to signal-style primitives. For grouped state transitions, `createPulseStore` is closer to a unified async update model: mutate state inside the pulse, then let the pulse system publish the result.

## 5.2.3. Comparison with Other Approaches

This table is conceptual. It compares common state-management styles by their usual programming model.

| Concept            | Async Support | Safe Direct Mutation | Manual Clone | Returns New State | Functions Needed |
| ------------------ | ------------- | -------------------- | ------------ | ----------------- | ---------------- |
| Reducer            | No            | No                   | Yes          | Yes               | 1-2              |
| Mutation           | No            | No                   | Yes          | Yes               | 2                |
| Producer           | No            | Yes                  | No           | No                | 1                |
| Action + Mutation  | Yes           | No                   | Yes          | Yes               | 2                |
| Action + Producer  | Yes           | Yes                  | No           | No                | 2                |
| Signal             | No            | No                   | No           | No                | 1                |
| `createPulse`      | No            | No                   | No           | No                | 1                |
| `createPulseStore` | Yes           | Yes                  | No           | No                | 1                |

Read the table this way:

- `createPulse` is the closest Valyrian primitive to a signal: one reactive value, one writer, immediate notification.
- `createPulseStore` is the more distinctive Valyrian model: one pulse method can handle state mutation, async work, batching, and early publication through `this.$flush()`.
- Reducer/action-style models usually split one async flow across multiple conceptual pieces; pulse stores keep that flow inside one named method.

Here, `Safe Direct Mutation` means the model is designed for you to mutate working state directly inside its normal update flow without returning a new state object and without side effects.

## 5.2.4. The Runtime Model Behind Pulses

Pulses subscribe by read, not by declaration.

### Reads inside render and effects

When you read a pulse or a pulse store state property:

- inside a render path, the current vnode is subscribed;
- inside `createEffect(...)`, the current effect is subscribed.

That is why components re-render only when they actually read the pulse, and why effects re-run only after they touch a pulse during execution.

For DOM subscriptions, the runtime records the vnode DOM node and calls `updateVnode(...)` for that subtree when the subscriber fires. The subscription is removed automatically once that DOM node is detached.

### Writes and notification timing

`createPulse` writes are immediate:

```ts
const [count, setCount] = createPulse(0);

setCount((current) => current + 1);
```

If the value actually changed, `write(...)` updates the stored value and runs subscribers right away in the same call stack.

`createPulseStore` behaves differently:

- each pulse method works against a mutable working copy of the store state;
- the committed public state stays immutable outside pulse methods;
- subscriber delivery is debounced with `setTimeout(..., 0)`.

That debounce means several store pulses called in the same tick collapse into one subscriber pass. The tests in `test/pulse-store_test.tsx` cover both the coalesced case and the later flush on the next tick.

### Async pulses and intermediate state

Async store pulses keep the same working copy alive until the returned promise settles. By default, subscribers only see the committed result when that async pulse finishes.

If you need an intermediate state such as `loading = true` before the async work finishes, call `this.$flush()` from inside the pulse method:

```ts
const store = createPulseStore(
  { todos: [], loading: false },
  {
    async fetchTodos(state) {
      state.loading = true;
      this.$flush();

      const response = await fetch("/api/todos");
      state.todos = await response.json();
      state.loading = false;
    }
  }
);
```

`$flush()` commits the current working copy early and schedules the normal debounced subscriber pass. It does not finish the pulse method; later mutations in the same async pulse can still commit again when the promise resolves.

## 5.2.5. `createPulse`

```ts
import { createPulse } from "valyrian.js/pulses";

const [count, setCount, runSubscribers] = createPulse(0);
```

Returned tuple:

1. `read()`
2. `write(newValue | updater)`
3. `runSubscribers()`

Use it this way:

- `read()` registers the current vnode or effect as a subscriber when one exists.
- `write(...)` only notifies when `hasChanged(...)` reports a real change.
- `runSubscribers()` is the escape hatch: it runs the current subscribers immediately even if the value did not change.

Use `runSubscribers()` only when you need to force recomputation without a value change. It bypasses the normal "only notify on change" rule and forces subscribed effects or vnode subtree updates to run again.

## 5.2.6. `createEffect`

```ts
import { createEffect, createPulse } from "valyrian.js/pulses";

const [count, setCount] = createPulse(0);

const dispose = createEffect(() => {
  console.log("count", count());
});

setCount(1);
dispose();
```

Behavior:

- the effect runs immediately once;
- any pulse or pulse-store state read during that run subscribes the effect;
- later notifications re-run the same effect function.

Warning: `createEffect()` does return a function, but the current implementation does not remove pulse subscriptions from the pulse or store that captured the effect. Outside vnode-specific usage it returns a no-op disposer, and even in vnode context it only removes an internal wrapper from a local set. Treat it as a limited disposer, not as a full unsubscribe contract.

If you need teardown tied to component lifecycle, prefer the runtime lifecycle helpers documented in [./3.1-runtime-core.md](./3.1-runtime-core.md).

## 5.2.7. `createPulseStore`

```ts
import { createPulseStore } from "valyrian.js/pulses";

const store = createPulseStore(
  { count: 0 },
  {
    increment(state) {
      state.count += 1;
    }
  }
);
```

Core rules:

- `store.state` is read-only from the outside in the immutable variant.
- pulse methods receive a mutable working copy as their first argument.
- nested pulses are batched through the same working copy and do not notify subscribers until the outer pulse commit point.
- subscriber delivery is debounced, so multiple calls in the same tick typically produce one subscriber pass.

This is why a store pulse is a better fit than multiple loose pulses when several properties should commit as one logical transition.

## 5.2.8. Advanced Store Helpers and Events

### `this.$flush()`

Inside a pulse store method, `this.$flush()`:

- commits the current working state early;
- keeps the pulse method alive;
- schedules the same debounced subscriber notification used by normal store commits.

Use it when an async pulse should reveal an intermediate state before it completes.

### `pulse` event timing

Store objects expose:

- `store.on(event, callback)`
- `store.off(event, callback)`

The main event documented in the implementation is `pulse`, and the callback receives `(pulseName, args)`.

Timing matters:

- for sync pulses, the `pulse` event fires after the pulse method returns;
- for async pulses, it fires after the returned promise resolves;
- it still fires even when the state did not change.

Because store subscriber updates are debounced, the `pulse` event is emitted before the debounced subscriber pass runs. Use `pulse` events for instrumentation or workflow hooks, not as your primary UI reactivity mechanism.

## 5.2.9. `createMutableStore`

`createMutableStore` keeps the same pulse API but disables the immutability guard on `store.state`.

Warning: changes made outside pulse methods do not trigger re-render notifications.

Use it only when you explicitly want direct mutation semantics. If you want the safer reactive path where public reads stay immutable and UI updates keep flowing through pulses, stay with `createPulseStore`.

## 5.2.10. Interaction with Delegated Events

Pulse stores participate directly in Valyrian's delegated event runtime.

When a pulse store method runs during an active delegated event, the implementation calls `current.event.preventDefault()` and `current.event.stopImmediatePropagation()` before executing the pulse body. In practice, that means the pulse opts out of the runtime's automatic post-handler renders and lets the pulse/store subscription system control when UI updates appear.

Operationally:

- a plain `createPulse` write notifies its subscribers immediately;
- a `createPulseStore` write schedules a debounced subscriber pass;
- `this.$flush()` lets an async store pulse expose an intermediate state on that debounced path before the pulse finishes.

This matters when mixing pulses with event handlers:

- if you call a store pulse from a delegated handler, do not expect the runtime's normal automatic rerender timing described in [./3.1-runtime-core.md](./3.1-runtime-core.md) to stay in charge;
- if you need exact control over what the user sees during an async interaction, decide whether the timing should come from pulse notifications, `this.$flush()`, or an explicit runtime `update()` path;
- if you combine `preventDefault()`, async handlers, and pulse stores, treat render timing as an explicit design choice, not an implementation detail.

Quick checklist:

- Use `createPulse` when one value can notify immediately on change.
- Use `createPulseStore` when several fields should move together and you may need debounced commits or `$flush()` for intermediate async state.
- Use `createMutableStore` only when direct mutation is more important to you than automatic reactive safety.
