# 4.2. Data Fetching and Async Workflows (Hub)

This chapter is a hub for networked and async modules used in SPA flows.

These modules layer on top of the base runtime. For the exact delegated event semantics that govern automatic rerenders, async handler timing, and `event.preventDefault()` behavior, see [./3.1-runtime-core.md](./3.1-runtime-core.md).

## Choose the Right Async Tool

| Need | Start here | Why |
| --- | --- | --- |
| Simple request/response call | [./4.2.1-request.md](./4.2.1-request.md) | Small fetch wrapper with serialization, parsing, plugins, and scoped clients. |
| Async rendering boundary | [./4.2.2-suspense.md](./4.2.2-suspense.md) | Handles fallback/error UI while async children resolve. |
| Cancellable user action | [./4.2.3-tasks.md](./4.2.3-tasks.md) | Adds run strategies, cancellation, and task state around async work. |
| Cached server state | [./4.2.4-query.md](./4.2.4-query.md) | Adds query/mutation handles, cache reuse, dedupe, and invalidation. |
| Connectivity-aware UI or network listeners | [./4.2.5-network.md](./4.2.5-network.md) | Tracks online/offline state and connection quality signals. |
| Offline-capable mutation queue | [./4.2.6-offline.md](./4.2.6-offline.md) | Persists operations and retries them when the network returns. |

Choosing an async module is a separate decision from the base delegated event rerender contract. Pick the module by data-flow need, then use [./3.1-runtime-core.md](./3.1-runtime-core.md) for the exact rules behind automatic updates, async handler timing, and `event.preventDefault()`.

Use these pages in order so each layer builds on the previous one.

## Common Beginner Mistakes

1. Using cache/invalidation before confirming plain request flows.
2. Skipping stable `Suspense` keys.
3. Assuming offline retry drains the queue in one sync pass.
4. Forgetting explicit cleanup (`off()`/`destroy()`) for long-lived listeners.
