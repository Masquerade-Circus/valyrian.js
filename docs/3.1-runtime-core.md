# 3.1. Runtime Core API (`valyrian.js`)

This page documents the runtime core behind the base `valyrian.js` package.

The same mount, update, lifecycle, and hydration rules described here power both browser execution and server-side rendering in Node.js. Read this page as the reference for the shared runtime model, then layer server-specific behavior from chapter 7 on top of it.

Use [./3-the-essentials.md](./3-the-essentials.md) for the learning path, then use this page for exact API behavior.

## VNode Construction Contract

Valyrian exposes two first-class vnode authoring surfaces:

- manual hyperscript through `v(...)` from `valyrian.js`
- automatic TSX/JSX runtime helpers through `valyrian.js/jsx-runtime` and `valyrian.js/jsx-dev-runtime`

Both surfaces build `Vnode` instances directly and share the same runtime contract.

## Render Lifecycle APIs

### `mount(dom, component)`

`mount` accepts:

1. A selector string or a DOM element as `dom`.
2. A component function.
3. An object or instance with `view()`.
4. A vnode component.
5. A plain value (wrapped into a trivial render function).

On mount, Valyrian hydrates the current container DOM into a vnode tree and then runs the normal patch cycle.

Returns:

- In Node.js, `mount()` returns the rendered HTML string for the mounted app.
- In browsers, `mount()` returns `""` after the patch completes.

Container resolution behavior:

- In browsers, a string container (for example `"body"` or `"#app"`) is resolved with `document.querySelector(...)`.
- In Node.js, a string container creates a fresh element via `createElement(...)` for that mount call.

If a browser selector does not resolve to an element, mounting cannot continue because there is no DOM root to hydrate.

The Node.js behavior is useful in SSR request handlers because each render call works with an isolated container element.

```tsx
import { mount } from "valyrian.js";

const App = () => <h1>Hello</h1>;
mount("body", App);
```

Mounting by selector or element is still supported for partial integration scenarios, but the default Valyrian architecture is app-owned `body`.

### `update()`

Runs a patch pass for the mounted app.

Returns:

- In Node.js, `update()` returns the mounted root HTML string after the patch.
- In browsers, `update()` returns `""`.

Use it when state changes happen outside delegated event handlers, when you call `preventUpdate()` and want manual control of renders, or when you want to force an intermediate state at a specific time. For the full delegated event timing contract, use the section below.

If a delegated handler returns a promise, Valyrian runs an automatic update after the synchronous part of the handler and another one when the promise settles. If the handler calls `preventUpdate()`, those automatic updates are skipped and you must call `update()` manually for each UI transition.

## Delegated Event Runtime Contract

Valyrian uses delegated DOM listeners for function-valued event props such as `onclick`, `oninput`, and `onsubmit`.

### Root listener registration

- Event listeners are registered on the mounted root DOM node (`mainVnode.dom`), not on each matching child element.
- Registration is lazy per event name: the runtime adds one root listener the first time it patches a function-valued prop for that event name.
- The listener set is cleared on `unmount()`, and also before switching the mounted app to a different container.

### Dispatch path and handler selection

When a delegated event fires, the runtime:

1. Stores the active event in `current.event`.
2. Starts at `event.target`.
3. Walks upward through ancestor DOM nodes.
4. Checks each vnode for a matching handler name such as `onclick`.
5. Runs the first matching handler and stops the walk.

Only the first matching delegated handler runs for that event dispatch.

### Update timing for sync and async handlers

- After the matching handler returns, Valyrian runs `update()` immediately unless `preventUpdate()` marked that event as update-prevented.
- If the handler returned a thenable, Valyrian schedules another `update()` when that promise settles, both on resolve and reject.
- This means async handlers have up to two automatic render points: one after the synchronous phase, and one after promise settlement.

Operationally:

- Synchronous mutations inside the handler are reflected by the immediate update.
- Mutations that happen after an `await` are reflected by the promise-settlement update, as long as `preventUpdate()` did not mark that event first.

### `preventUpdate()`

`preventUpdate()` only controls Valyrian's automatic rerender points for the active delegated event.

- `preventUpdate()` is event-scoped.
- `preventUpdate()` is a no-op when there is no active delegated event.
- If you call it during the synchronous phase of a delegated handler, Valyrian skips both the immediate update and the later promise-settlement update for that same event.
- Once you opt out of the automatic render path with `preventUpdate()`, you are responsible for calling `update()` manually for each UI transition you want to show.

Practical consequence:

- Use `preventUpdate()` when the default render timing is not the timing you want the user to see.
- Keep the default automatic render path when the built-in sync and promise-settlement updates already match your UI flow.

Pulse-store exception:

- When a `createPulseStore` pulse runs inside an active delegated event, the pulse runtime can take over render timing before the normal delegated-event auto-update path runs.
- In the current implementation, it does that by calling `preventUpdate()` and `current.event.stopImmediatePropagation()` before executing the pulse body.
- In that situation, rely on pulse notifications, debounced store delivery, and `this.$flush()` for timing instead of the default delegated-event auto-render sequence.

### Operational Notes

- Browser runtime mounting is singleton-style: `mount()`, `update()`, and `unmount()` operate on one global `mainComponent`/`mainVnode` pair at a time.
- Calling `mount()` again replaces that mounted app reference with the new component/container pair.
- If the new mount uses a different container, Valyrian removes delegated root listeners from the previous root before continuing.
- `unmount()` renders a null app, removes delegated listeners, and resets mounted runtime state.
- Remounting after `unmount()` creates a fresh mounted root contract; previous delegated listeners do not stay attached.

### `unmount()`

Unmounts the current app, cleans listeners/resources, and clears delegated event bindings.

Returns:

- In Node.js, `unmount()` returns the final HTML string produced by rendering the null app.
- In browsers, `unmount()` returns `""`.

## Component Lifecycle Helpers

Use these inside a component render path:

- `onCreate(fn)`
- `onUpdate(fn)`
- `onCleanup(fn)`
- `onRemove(fn)`

These helpers throw if called outside component execution.

Practical timing model:

- `onCreate`: runs when the component appears in the tree.
- `onUpdate`: runs on later patches while the component stays in the tree.
- `onCleanup`: runs before each patch of that vnode subtree and before subtree detach.
- `onRemove`: runs after detach, when the component subtree is actually removed.

Contract details:

- `onCreate(fn)` accepts `() => void | (() => void) | Promise<void>`.
- If `onCreate` returns a cleanup function synchronously, Valyrian registers it immediately.
- If `onCreate` returns a promise, Valyrian schedules `debouncedUpdate()` after resolve or reject.
- If that promise resolves to a function, that async cleanup is invalid and is ignored.
- SSR does not wait for async `onCreate` work before producing the initial HTML. If the first HTML must already contain fetched data, prefetch it before calling `render()`/`mount()` and pass it in as initial state.

```tsx
import { onCreate, onRemove } from "valyrian.js";

let disposed = false;
const UserPanel = ({ api }) => {
  onCreate(async () => {
    const user = await api.get("/api/user");

    if (disposed) {
      return;
    }

    console.log("Loaded user", user.id);
  });

  onRemove(() => {
    disposed = true;
  });

  return <div>User panel</div>;
};
```

## `debouncedUpdate(timeout?)`

Debounces rendering. Default timeout is `42ms`.

If called from an active event context, it calls `preventUpdate()` for that event before scheduling the debounced update.

```tsx
import { debouncedUpdate } from "valyrian.js";

const SearchBox = {
  term: "",
  onInput(event: Event) {
    this.term = (event.target as HTMLInputElement).value;
    debouncedUpdate(80);
  },
  view() {
    return <input value={this.term} oninput={(event) => this.onInput(event)} />;
  }
};
```

## Raw HTML: `trust(html)`

`trust` parses an HTML string into vnode-compatible children and marks that HTML as intentionally raw.

```tsx
import { trust } from "valyrian.js";

const Rich = ({ html }) => <article>{trust(html)}</article>;
```

Only use trusted or sanitized HTML. `trust(html)` is an explicit escape hatch from Valyrian's normal text escaping; in SSR it renders the provided HTML directly. Do not pass user-controlled strings unless your application has sanitized them first.

## `v-model` Contract

`v-model` is a built-in directive with mode-specific behavior.

Rules:

1. The control must have `name`.
2. Existing control handler (`oninput`/`onclick`) is preserved and runs after model sync.

Reader note: for `select[multiple]`, this implementation listens on `onclick` and supports additive selection with `Ctrl`.

Behavior by control type:

- `input` (default types): syncs `value` with `oninput`.
- `input[type=checkbox]`:
  - model array: toggle `value` membership.
  - scalar with `value` prop: toggle between `value` and `null`.
  - scalar without `value`: toggle boolean.
- `input[type=radio]`: checked when `model[name] === input.value`.
- `select` single: sets selected option from model.
- `select[multiple]`: model is array, selection handler runs on `onclick`.
- `textarea`: vnode children mirror model value.

```tsx
const state = { tags: [], newsletter: false, role: "user", bio: "" };

const Form = () => (
  <form>
    <input type="checkbox" name="newsletter" v-model={state} />

    <select multiple name="tags" v-model={state}>
      <option value="ts">TypeScript</option>
      <option value="ssr">SSR</option>
    </select>

    <textarea name="bio" v-model={state} />
  </form>
);
```

## Hydration Behavior

At mount time, existing DOM is converted to vnode structure (including text and attributes), then patching continues from that structure.

This allows SSR HTML and pre-existing markup to become interactive without replacing the full container tree.
